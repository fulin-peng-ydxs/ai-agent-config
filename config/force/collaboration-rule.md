# AI 协作约束（通用红线）

## 回复用户时

- 使用简体中文，命令、代码、日志和文件原文除外。

## 准备改代码时

- 检查最近一次 `git pull` 时间；超过 2 小时执行 `git pull --rebase`，成功后更新时间再改。
- `git pull` 冲突或失败时，先解决同步问题。
- 最近一次 `git pull` 操作时间：2026-07-12 01:22:04 CST
- 注意：内容要写入到项目的AGENTS.md，此文档这里为`模版`

## 需要先确认时

- 开发计划涉及技术方案、交互设计或关键实现决策时，先给推荐选项和理由，确认后再开发。
- 业务规则不确定时，先确认，不得写死。

## 写项目文档时

- 默认放 `./agent-works/`；具体需求、功能或模块放 `./agent-works/{feature-slug}/`。
- 同一功能的开发计划、开发留痕、业务核查、SQL、测试记录等复用同一目录；已有相关目录优先复用。
- 不在项目根目录生成零散 Markdown，除非用户明确指定。

## 命名功能目录时

- `{feature-slug}` 表示功能或需求，使用小写 kebab-case，并表达业务含义。
- 禁止 `feature`、`update`、`code-change`、`development-trace` 等泛名。
- 同一功能已有 slug 必须沿用；中文业务名不稳定时，先建议 slug 并等待确认。

## 写代码时

- 新增或修改前先参考同类实现，优先复用现有的能力。
- 保持项目现有布局、UI、目录、命名、分层、返回结构、异常处理、日志、权限校验和配置方式；偏离时说明原因。
- 改动清晰、克制、易扩展；避免无必要的新依赖、新抽象、过度封装和硬编码；公共逻辑按既有方式沉淀。
- 关键代码必须有必要的代码注释说明；

## 写注释时

- 优先用命名和结构表达意图，不写复述代码的低价值注释。
- 复杂业务规则、状态流转、权限边界、兼容逻辑、性能取舍、异常兜底、外部系统约束必须注释。

## 处理性能和资源时

- 考虑数据量、调用频率、事务范围、查询复杂度、缓存影响、资源释放和重复计算。
- 避免 N+1 查询、大对象一次性加载、重复远程调用、无限增长集合、阻塞式长任务和资源泄漏。

## 功能测试时

- 所有代码改动先做代码自审，再按改动类型执行最低测试。
- 前端不涉及点击类交互的小修改，至少执行静态编译。
- 前端新开发页面或涉及点击类交互时，必须执行构建，并用浏览器自动化操作验证关键交互。
- Java 后端改动至少执行项目编译测试。

## 开发页面时

- 仅当任务影响页面、组件、样式、布局、主题、设计 token 或交互表现时，使用前端设计能力。
- 涉及界面风格、视觉统一、品牌一致性或设计优化时，按需读取 `DESIGN.md` 相关部分；需要高质量界面生成或避免模板化输出时，可启用 `frontend-design`。
- 后端、接口、数据库、脚本、测试、构建、部署、性能、安全等非 UI 任务，不启用 `frontend-design`，不读取 `DESIGN.md`。

## 写数据库SQL时

- 涉及数据库变更、初始化数据、字典、菜单、权限、配置或迁移 SQL 时，写清 SQL 文件路径、用途、执行时机、影响范围、回滚方式或不可回滚原因。
- SQL 默认放 `./agent-works/{feature-slug}/{sql-purpose}.sql`；
- SQL 文件必须有注释、包括总体、字段、执行语句的注释，说明用途、影响范围、前置条件、回滚方式或不可回滚原因”。

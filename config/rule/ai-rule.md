## 工作约束红线

### 001 修改代码前先同步远端
- 触发：准备修改代码前
- 要求：检查本规则记录的最近一次 `git pull` 时间；超过 2 小时先执行 `git pull --rebase`，成功后更新时间，再改代码
- 失败：`pull` 冲突或失败时，先处理同步问题，再继续
- 记录：最近一次 `git pull` 操作时间：未执行

### 002 AI 文档默认目录
- 默认目录：用户未指定路径时，AI 生成的项目文档默认放到 `./agent-works/`
- 功能目录：文档属于具体需求、功能或模块时，优先放到 `./agent-works/{feature-slug}/`
- 目录复用：同一功能的开发计划、开发留痕、业务核查、SQL、测试记录等必须复用同一目录；若已有相关目录，优先复用
- 根目录限制：不要在项目根目录生成零散 Markdown，除非用户明确指定

### 003 slug 命名与复用
- 命名：slug 统一使用小写 kebab-case，并表达业务含义
- 禁止：禁止使用 `feature`、`update`、`code-change`、`development-trace` 等泛名
- 复用：已有开发计划、留痕、核查、SQL 或目录使用过的 slug，后续同一功能必须沿用
- 确认：中文业务名不稳定时，先给出建议 slug，并等待用户确认
- 占位：`{feature-slug}` 表示功能或需求，`{plan-slug}` 表示开发计划主题，`{sql-purpose}` 表示 SQL 用途

### 004 语言与会话命名
- 语言：除命令、代码、日志、文件原文外，所有会话交互必须使用简体中文
- 命名：新会话首次回复后，应将会话名称重命名为中文；格式为 `{业务名称-动作}`，简短清晰，例如 `技能优化-规则补充`

### 005 通用编码约束
- 注释：优先用清晰命名和结构表达意图；复杂业务规则、状态流转、权限边界、兼容逻辑、性能取舍、异常兜底、外部系统约束必须注释
- 注释禁区：禁止复述代码的低价值注释；SQL 文件必须有文件级注释，说明用途、影响范围、前置条件和回滚方式或不可回滚原因
- 复用与风格：新增或修改前必须先参考项目已有同类实现，优先复用现有能力，并保持目录、命名、分层、返回结构、异常处理、日志、权限校验和配置方式一致；偏离既有风格时必须说明原因
- 性能与资源：必须考虑数据量、调用频率、事务范围、查询复杂度、缓存影响、资源释放和重复计算；避免 N+1 查询、大对象一次性加载、重复远程调用、无限增长集合、阻塞式长任务和资源泄漏
- 可维护性：改动应清晰、克制、易扩展；避免无必要的新依赖、新抽象、过度封装和硬编码；公共逻辑按项目既有方式沉淀；不确定的业务规则不得写死，必须显式确认

### 006 前端 UI 设计能力按需使用
- 启用：只有任务直接影响前端页面、组件、样式、布局、主题、设计 token 或交互表现时，才使用前端设计相关能力
- 可用：涉及界面风格、视觉统一、品牌一致性或设计优化时，可读取根目录 `DESIGN.md`；涉及高质量界面生成或需要避免模板化输出时，可启用 `frontend-design` 技能
- 禁用：后端、接口、数据库、脚本、测试、构建、部署、性能、安全及其他非 UI 任务，不启用 `frontend-design`，也不读取根目录 `DESIGN.md`
- 范围：`DESIGN.md` 仅按需读取与当前改动相关的部分，不作为默认上下文

## 开发计划约束

### 001 SQL 变更说明
- 适用：开发计划涉及数据库变更、初始化数据、字典、菜单、权限、配置或迁移 SQL
- 要求：必须明确 SQL 文件路径、用途、执行时机、影响范围、回滚方式或不可回滚原因
- 默认路径：`./agent-works/{feature-slug}/{sql-purpose}.sql`
- 禁止：不得只写“需要执行 SQL”

### 002 Codex 计划文件归档
- 触发：Codex 生成开发计划，且用户确认按该计划执行开发后，在开发完成时归档计划文件
- 归档：归档到 `~/.codex/plans/{plan-slug}.md`
- 命名：`{plan-slug}` 使用小写 kebab-case，并表达开发计划主题
- 仓库同步：仅用于 Codex 自身计划文件归档；不要求同步到项目仓库，除非用户明确要求

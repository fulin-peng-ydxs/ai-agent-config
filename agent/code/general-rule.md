# 通用代码开发要求

本文件是 AI 开发时默认读取的通用规则。若用户额外指定项目规则文件，应同时遵守；若规则冲突，以用户本次明确要求和更具体的项目规则优先。

## 需求与范围要求

- 开发前先理解本次目标、输入输出、影响范围和验收点，不清楚时先确认，不自行扩展需求。
- 每个代码改动都应能对应到本次需求；避免顺手重构、顺手修复无关问题或改动未授权文件。
- 涉及既有流程时，必须考虑成功路径、失败路径、空值/空集合、权限限制、重复操作、历史数据和兼容场景。
- 前后端或多模块联动时，必须同时关注调用方、实现方、数据结构、错误语义和边界行为是否一致。

## 复用与风格要求

- 新增或修改前先查找同类实现，优先复用已有工具、组件、封装、模型、常量、异常处理、响应结构和交互模式。
- 同类能力应放到项目已有分层和业务域中，不新增孤立目录、平行体系或重复抽象。
- 命名、目录、参数组织、返回结构、日志、异常、状态处理和 UI 交互、注释等应与项目现有风格一致。
- 确需新增公共能力时，只提取真实复用点，保持抽象最小，并避免引入无必要依赖或全局状态。

### 前端设计上下文使用规则

仅在任务直接涉及前端界面或交互改动时，才启用前端设计相关能力。

符合以下情况时，可启用 `frontend-design` 技能，并按需读取 `DESIGN.md`：
- 新增或修改页面、组件、布局、样式、主题、设计 token
- 调整用户可见的视觉表现或交互表现
- 生成落地页、后台界面、表单、看板、导航、弹窗等前端界面
- 任务明确要求优化界面美观度、设计一致性、交互体验或品牌风格

不符合以下情况时，不要启用 `frontend-design` 技能，也不要读取 `DESIGN.md`：
- 后端、数据库、接口、脚本、测试、构建、部署、运维、安全、性能优化
- 不影响界面表现的逻辑修改或代码重构
- 与前端设计无关的文档、配置或数据处理任务

启用前端设计相关能力时，优先级如下：
1. 先判断当前任务是否真的涉及前端 UI/UX
2. 若仅为普通前端实现，可使用现有项目样式和组件，不必读取 `DESIGN.md`
3. 若任务涉及界面风格、视觉统一、设计质量、品牌一致性，再读取 `DESIGN.md`
4. 若任务明确要求高质量界面生成、设计增强或避免通用模板感，可启用 `frontend-design` 技能

读取 `DESIGN.md` 后：
- 只提取与当前任务直接相关的约束，避免全文复述
- 优先遵循其中的颜色、字体、间距、组件和交互规则
- 如果项目现有实现与 `DESIGN.md` 冲突，默认优先保持现有项目风格，除非任务明确要求做 UI 统一或设计重构

## 注释要求

- 类、字段、方法、核心逻辑和复杂分支应根据可维护需要补充清晰注释，说明职责、业务语义、约束、输入输出和影响范围。
- 注释重点解释业务背景、设计意图、关键判断依据、边界处理和兼容原因，不逐句翻译代码。
- 状态值、枚举、金额/数量/时间单位、外部协议字段、历史数据兜底等容易误用的内容必须写明业务含义。
- 临时兼容、异常绕行、历史修正、外部系统特殊约束必须说明触发条件、保留原因和后续处理建议。
- 旧注释与新逻辑不一致时必须同步更新；避免“查询数据”“保存信息”这类无信息量注释。
- SQL 文件必须有文件级注释，说明用途、影响范围、前置条件和回滚方式或不可回滚原因。

## 数据、异常与安全要求

- 不随意改变字段含义、状态值、枚举值、时间格式、精度、分页语义、接口路径和响应结构。
- 空值、缺失配置、接口失败、权限不足、重复提交、并发冲突等边界应沿用项目已有处理方式。
- 异常处理要可追踪、可定位，不吞异常；日志应保留排查信息，但不得输出账号、密码、token、密钥、连接串等敏感值。
- 认证、权限、数据范围、租户、部门、角色等逻辑必须复用已有机制，避免绕过统一校验。
- 文件、URL、HTML、SQL、脚本、富文本、上传下载等输入输出场景，应考虑注入、越权、路径穿越和内容污染风险。

## 性能与稳定性要求

- 实现时必须考虑数据量、调用频率、资源生命周期和失败重试，避免明显性能退化。
- 避免循环中重复查询/请求/IO、重复创建大对象、重复计算、无界缓存、无界集合和一次性加载超大数据。
- 前端或交互层避免无意义重渲染、深层监听、高频轮询、未清理定时器/监听器和重复请求。
- 后端或服务层避免未关闭资源、无限分页、长事务滥用、阻塞关键线程和不可控异步任务。

## 可维护与扩展要求

- 改动应小而清晰，优先解决当前问题；不要为了“看起来更完整”引入复杂结构。
- 公共逻辑、配置项、魔法值、常量和类型应按项目既有方式沉淀，避免散落在业务代码中。
- 新逻辑应便于后续定位、测试和扩展；关键行为不要隐藏在难以发现的副作用里。
- 删除或替换既有逻辑时，必须确认调用方、兼容行为、默认值、数据迁移和回滚影响。
